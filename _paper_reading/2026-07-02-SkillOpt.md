这篇文章主要解决的是：现有 agent skill 的自我修订过程过于松散，常常出现大幅语义跳跃、无约束改写和不稳定更新，导致 skill 不能在反馈中稳定进步。

作者提出将 skill 文档本身视为 frozen agent 的外部可训练状态，并类比深度学习中的训练优化机制：skill document 对应参数，trajectory-derived edit direction 对应梯度方向，edit budget 对应学习率，held-out selection gate 对应验证集检查，batch / minibatch / schedule / gate 对应稳定训练设置。

基于此，作者构建了 SkillOpt，一个可控的 text-space optimizer，用来离线训练可复用的 agent skill。结果显示，SkillOpt 在多个 benchmark、模型和执行 harness 上都取得了显著提升，说明这种“有边界、有验证、有负反馈记忆”的渐进式 skill 优化比松散的自我改写更稳定有效。

## 图 1：将 skill 优化类比为深度学习训练

![SkillOpt Figure 1](https://raw.githubusercontent.com/Bunnybeibei/jiabeicheng.github.io/master/images/SkillOpt-fig1.png)

Fig. 1 说明了 SkillOpt 的核心类比：把 skill document 当作可训练参数，把从 trajectory 中总结出的 edit direction 当作梯度方向，把 edit budget 当作学习率，把 held-out selection gate 当作验证集检查。

这个类比非常关键，因为它把“让 LLM 自己改 prompt / skill”从一个松散的文本改写问题，转化为一个有训练状态、有步长控制、有验证门控、有负反馈记忆的优化问题。

## 对比方法

作者比较了以下 baselines：

* no skill
* human skill
* one-shot LLM skill
* Trace2Skill
* TextGrad
* GEPA
* EvoSkill

其中，no skill、human skill 和 one-shot LLM skill 分别对应无适应、人工书写和一次性生成的对照；TextGrad 和 GEPA 属于 prompt optimization；Trace2Skill 和 EvoSkill 更接近 skill distillation / skill evolution。作者也因此将 baseline 分成 prompt 的优化与 skill 的优化两类。

## 图 2：SkillOpt 的训练流程

![SkillOpt Figure 2](https://raw.githubusercontent.com/Bunnybeibei/jiabeicheng.github.io/master/images/SkillOpt-fig2.png)

作者将数据划分为 train / selection / test 三部分：

* **train split**：用来产生 rollout trajectories，作为修改 skill 的证据。
* **selection split**：用来做 validation gate，决定 candidate skill 是否被接受。
* **test split**：被锁定，只用于最终报告性能。

这个划分让 skill 的更新不再只依赖训练样本上的即时反馈，而是必须经过 held-out selection split 的检验。这也是 SkillOpt 稳定性的主要来源之一。

---

## 关键机制

### Rollout Evidence

Rollout Evidence 指当前 skill 在多个任务样本上的执行轨迹和得分。作者认为，较大的 rollout batch 可以暴露更稳定的共性错误，较小 batch 更新更快但噪声更大。

### Minibatch Reflection

Optimizer model 将 rollout trajectories 按成功和失败分开，并进一步划分为 reflection minibatch。这样做是为了从多个样本中总结可复用的程序性错误，而不是被单个失败样本牵着走。

### Bounded Text Updates

Optimizer 可能从一批 rollout 里总结出很多条修改建议，但这一轮真正写进 skill 文档的修改数量是有限的。这个 bounded edit budget 类似学习率控制，用来避免 skill 文档发生大幅语义漂移。

### Validation Gate and Rejected-Edit Buffer

每个 candidate skill 都会在 selection split 上评估。如果它严格提升当前 selection score，就成为新的 current skill；如果它同时超过历史最好分数，就更新为 `best_skill.md`。如果 candidate skill 没有通过 gate，被拒绝的 edits 会进入 rejected-edit buffer，作为后续优化时的负反馈。

### Epoch-wise Slow / Meta Update

每个 epoch 结束时，SkillOpt 会用 previous epoch skill 和 current epoch skill 在相同训练样本上比较效果，并把样本分成 improvements、regressions、persistent failures 和 stable successes。这个慢速/meta 更新用于总结哪些修改真的有帮助，哪些修改带来了退化，哪些失败一直存在。

### Harness-Agnostic Deployment

这里的重点是 harness 通用。SkillOpt 通过轻量 adapter 将 skill 注入不同执行环境，包括 direct chat、Codex harness 和 Claude Code harness，并统一返回 scored trajectories。

最终部署的是同一个 `best_skill.md`，不需要修改目标模型权重，也不会在推理时额外调用 optimizer model。这也是它能够做 cross-model、cross-harness 和 cross-benchmark transfer 的原因。

## 图 3：主结果

![SkillOpt Figure 3](https://raw.githubusercontent.com/Bunnybeibei/jiabeicheng.github.io/master/images/SkillOpt-fig3.png)

Fig. 3 显示，SkillOpt 在 6 个 benchmark、7 个 target models 和 3 种 execution harness 上取得了系统性优势，在 52/52 个评估单元中达到 best 或 tied-best。

这说明 SkillOpt 不只对大模型有效，小模型也能受益。在 GPT-5.5 direct chat 下，平均准确率相对 no skill 提升 +23.5 points；在 Codex / Claude Code 这种 agent harness 中也有效，分别提升 +24.8 和 +19.1 points。

## 图 4：迁移能力

![SkillOpt Figure 4](https://raw.githubusercontent.com/Bunnybeibei/jiabeicheng.github.io/master/images/SkillOpt-fig4.png)

Fig. 4 的 transfer 实验证明，优化后的 skill 可以跨模型、跨 harness、跨相近 benchmark 迁移。

这部分很重要，因为它说明 SkillOpt 训练出的不是某个模型或某个执行环境中的过拟合文本，而是可以被复用的程序性策略。换句话说，SkillOpt 优化的是 skill 文档中可迁移的执行知识。

## 图 5：消融实验

![SkillOpt Figure 5](https://raw.githubusercontent.com/Bunnybeibei/jiabeicheng.github.io/master/images/SkillOpt-fig5.png)

消融实验表明，以下机制是稳定提升的关键：

* **bounded edit budget**：每一步只允许有限数量的 add / delete / replace edits，避免 skill 文档大幅漂移。
* **validation gate**：candidate skill 必须在 selection split 上严格提升才会被接受，tie 也会被拒绝。
* **rejected-edit buffer**：被拒绝的 edits 会作为负反馈，帮助后续 optimizer 避免重复犯错。
* **epoch-wise slow / meta update**：跨 epoch 总结哪些修改有帮助、哪些会造成退化、哪些失败一直存在。

这些机制共同回答了文章的核心问题：skill 自我改写不应该是无限制地“重新写一版”，而应该像训练模型一样，进行有步长、有验证、有记忆的稳定更新。

## 结果组织方式总结

这篇文章的结果组织很像一篇“优化器论文”：先用 Fig. 1 建立参数优化类比，再用 Fig. 2 给出训练流程，随后用 Fig. 3 证明主效果，用 Fig. 4 证明迁移性，最后用 Fig. 5 拆解哪些机制真正重要。

我觉得最值得学习的一点是：作者没有只说“LLM 可以自动改 skill”，而是把自我修订过程拆成了可控优化组件。这样一来，方法贡献不再只是一个 pipeline，而是一套可以分析、消融和迁移的训练策略。
